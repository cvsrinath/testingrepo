# KYP Platform Evolution - Major Product Scope Change

You have already built the current KYP prototype and should treat the attached document **"KYP UI Prototype - Current Implementation Snapshot (2026-06-15)"** as the authoritative baseline of the existing implementation.

Before making changes, carefully review that document because it accurately reflects the current routes, components, data models, mock services, and overall architecture.

---

# Product Direction Change

The product is no longer a standalone application called KYP.

KYP now becomes one module (or workspace/tab) inside a larger internal engineering operations platform.

The existing KYP functionality should remain available, but it should be reorganized as part of a broader application that contains multiple operational domains.

Think of the new product as an internal engineering and workforce operations portal with several major functional areas, where KYP is only one of them.

---

# New Application Structure

The application should evolve from:

Current:

* KYP

To:

* Engineering Operations Platform

  * KYP (Know Your Performance)
  * Employee Onboarding
  * Infrastructure
  * Training Management
  * Stakeholder Reports (placeholder for future implementation)

KYP should remain fully functional but become one section of the larger platform.

---

# Primary Goal

Refactor the application structure so that it can support multiple major modules while maintaining a consistent navigation system, layout, routing strategy, shared design language, and reusable data architecture.

Do not think of the new modules as simple pages.

Each module should be designed as if it could eventually become as large and feature-rich as KYP.

---

# Architecture Expectations

## Create a Platform Shell

Introduce a top-level application shell that provides:

* Global navigation
* Module navigation
* Shared layout
* Shared page structure
* Shared filtering patterns where appropriate
* Consistent header and breadcrumb behavior

Suggested navigation:

* Dashboard / Home
* KYP
* Onboarding
* Infrastructure
* Training
* Stakeholder Reports

The exact UI can be refined, but the application should clearly feel like a multi-module platform rather than a standalone KYP product.

---

# KYP Module

The existing KYP implementation should be preserved as much as possible.

Existing routes should continue to work, but should be reorganized under a KYP module structure.

Examples:

Current:

* /teams
* /teams/[teamId]
* /employees/[id]
* /admin/scoring

Potential future structure:

* /kyp/teams
* /kyp/teams/[teamId]
* /kyp/employees/[id]
* /kyp/admin/scoring

A migration approach is acceptable.

Legacy routes may temporarily remain for compatibility.

Do not remove existing KYP functionality.

* Employment type

Data Team:

## Dashboard Concepts

# New Module 2: Infrastructure

Create an infrastructure visibility module.

Purpose:

Provide a company-wide view of infrastructure ownership and usage by teams.

This is not a DevOps monitoring system.

It is primarily an organizational infrastructure inventory and ownership view.

## Core Concepts

Infrastructure assets:

* Cloud projects/accounts
* Kubernetes clusters
* Databases
* Repositories
* CI/CD systems
* Monitoring systems
* Internal services
* Shared platforms

## Team Mapping

* Which teams own which infrastructure
* Shared infrastructure
* Dependencies
* Service ownership

## Suggested Views

### Infrastructure Directory

Searchable inventory of infrastructure assets.

### Infrastructure Detail

Asset information:

* Owner team
* Description
* Environment
* Criticality
* Dependencies
* Related systems

### Team Infrastructure View

For a team:

* Owned assets
* Consumed assets
* Shared resources

### Dashboard

Examples:

* Infrastructure by team
* Infrastructure by environment
* Critical systems
* Ownership coverage

Mock data is acceptable.

# Important Constraints

Do NOT implement:

* Authentication
* RBAC
* Real databases
* Real integrations
* Real onboarding systems
* Real cloud APIs
* Real training providers

Continue using mock services and prototype-level data patterns.

The goal is to create a scalable UI and application architecture that demonstrates workflows and future platform direction.
